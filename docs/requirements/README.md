# 요구사항 문서 작성 가이드

이 폴더는 제품 요구사항을 코드보다 먼저 정리하기 위한 기본 문서 세트입니다. 새 기능을 만들 때는 구현을 시작하기 전에 아래 문서에 요구사항을 먼저 적습니다.

## 작성 순서

1. `PRD.md`: 사용자의 문제, 목표, 범위, 성공 기준을 작성합니다.
2. `Use-Case.md`: 실제 사용자가 어떤 상황에서 어떤 흐름으로 기능을 쓰는지 작성합니다.
3. `IA.md`: 페이지, 메뉴, 정보 구조, 라우팅, 화면 간 이동을 작성합니다.
4. `ERD.md`: DB 테이블, 관계, PK/FK, 인덱스, 보존/삭제 정책을 작성합니다.
5. `TRD.md`: API, DB, 인증, 결제, 배포, 테스트 등 기술 구현 요구사항을 작성합니다.
6. `SEO.md`: Metadata, OG, JSON-LD, sitemap, robots, hreflang, GEO, GSC, GA4, Core Web Vitals 등 검색엔진·AI 검색 노출 요구사항을 작성합니다.
7. `../design-system/Design-System.md`: 화면을 만들 때 사용할 토큰, 컴포넌트, 상태, 접근성 기준을 작성합니다.

## 원칙

- 요구사항은 코드, 이슈, 채팅에 흩어두지 말고 이 폴더의 문서에 먼저 모읍니다.
- DB 스키마를 만들기 전에는 `ERD.md`에 엔티티, 관계, 키, 인덱스, 삭제/보존 정책을 먼저 적습니다.
- i18n 다국어 요구사항은 지원 언어, 기본 locale, 번역 범위, 언어 선택 UI, sitemap/hreflang 정책까지 PRD/IA/TRD에 함께 적습니다.
- 이 템플릿은 한국어를 기본으로 하고 영어를 상단 메뉴에서 선택할 수 있으며, locale과 번역 파일을 추가해 20개 국어 이상으로 확장할 수 있게 관리합니다.
- 결정되지 않은 내용은 `미정`으로 표시하고, 추측해서 구현하지 않습니다.
- 구현이 끝난 뒤 실제 동작과 문서가 달라졌다면 문서를 같이 갱신합니다.
- 공개 저장소에 올릴 수 없는 고객명, API key, 내부 URL, 개인정보는 적지 않습니다.
